## D002 Q-FIN Notes: Order Types

### What is it?

订单类型是你把交易意图交给市场撮合系统时使用的执行方式。它决定你优先要“立刻成交”、还是“控制价格”、还是“触发风险动作”。

| Order Type | 中文理解 | 核心交换 |
|---|---|---|
| Market order | 市价单 | 优先成交速度，牺牲价格控制。 |
| Limit order | 限价单 | 优先价格控制，牺牲成交确定性。 |
| Stop order | 触发单/止损触发 | 到达某个条件后才变成执行动作，常用于风险控制。 |

### Core mechanism

- Market order 会直接吃掉 order book 里对手方的挂单。买入 market order 会吃 ask，卖出 market order 会砸 bid。
- Limit order 是把你的价格条件挂进 order book。只有市场价格碰到你的条件、且排队轮到你时，才可能成交。
- Stop order 不是立刻执行，而是到达触发价后才变成 market order 或 limit order。

### How should I use it?

| 场景 | 应该优先考虑 | 原因 |
|---|---|---|
| 必须马上退出风险 | market order 或 stop-market | 成交优先，但要接受滑点。 |
| 想控制买入/卖出价格 | limit order | 价格优先，但可能不成交。 |
| 做 agent 自动交易 | 先定义订单类型权限 | 不同订单类型带来完全不同的风险。 |
| 回测策略 | 必须模拟订单类型 | 不能默认所有信号都按理想价格成交。 |

### Failure modes

- 错误 1：把 market order 当成“按屏幕价格成交”。实际是吃 order book，可能滑点很大。
- 错误 2：把 limit order 当成“必然成交”。实际可能挂在那里没人打到，或者排队排不到。
- 错误 3：把 stop order 当成“保证止损价成交”。stop-market 只能保证触发后尽量成交，不保证触发价成交。

### ROI rule

> Trading agent 不能只输出 buy/sell。它必须输出 order type、最大滑点、最大仓位、有效期、取消条件和异常停机规则。

## D002 PHYS Notes: State

### What is it?

State 是系统的可计算描述，不是现实本身的完整复制。

通俗说：state 是你用来预测和计算的“压缩模型”。它保留对当前问题有用的信息，丢掉大量你没有测、不能测、或当前模型不表达的信息。

### Core mechanism

在 classical thinking 里，我们容易以为：物体已经有确定的位置、速度、属性，只是我们还不知道。

在 quantum physics 里，state 更像一个“能产生测量结果概率的结构”。你不是直接拿到现实全部细节，而是拿到一个能告诉你“如果测这个 observable，可能得到什么结果”的数学对象。

### How should I use it?

| 场景 | 低水平理解 | 高水平理解 |
|---|---|---|
| 看 AI 输出 | 模型给了答案 | 这是在某个上下文 state 下的一次输出，不等于现实。 |
| 看市场价格 | 价格就是市场真相 | 价格是订单、流动性、参与者、时间窗口共同压缩出的状态信号。 |
| 看论文结论 | 论文证明了结论 | 论文是在特定假设、模型、测量方式下得到的状态描述。 |
| 做决策 | 我要真实答案 | 先问这个 state 包含什么、忽略什么、能预测什么、不能预测什么。 |

### Failure modes

- 错误 1：把 state 当现实本身。
- 错误 2：忘记 state 依赖测量和建模方式。
- 错误 3：拿一个场景下有效的 state 描述，迁移到另一个场景里强用。

### ROI rule

> 任何模型、数据、报告、价格、AI 回答，都先问：这是哪个系统的 state？由哪些观测生成？能预测什么？忽略了什么？边界在哪里？

## Cross-Domain Connection

Q-FIN 的订单类型告诉你：一个意图进入真实系统后，会被执行机制改变结果。  
PHYS 的 state 告诉你：一个现实对象进入模型后，会被描述方式压缩成某种状态。

共同判断力：

> 不要把“意图”当结果，不要把“描述”当现实。
