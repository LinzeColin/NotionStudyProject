## 为什么补强

用户明确反馈：核心问题“不知道答案”，需要详细、专业、全面、深度讲解这句话：

> RAG 可以理解为：系统先从你的文档、日志、SOP、设备手册、历史案例、工艺数据说明里找证据，再把证据交给 LLM 生成回答。它的价值不在于让模型“更聪明”，而在于让模型的回答有来源、可检查、可追责。

因此 D02 不应直接推进。D01 的资产已完成，但概念掌握需要独立复述校验。

## 一句话核心

RAG 不是把知识“塞进模型脑子里”，而是在模型回答前给它一份可追溯的证据包。LLM 负责组织、解释、推理和表达；RAG 负责把回答锚定到你的真实资料。

## 为什么不是“让模型更聪明”

LLM 的参数知识像压缩后的通用经验：它能说出很多合理的话，但它不天然知道你的企业文档、设备版本、最新 SOP、现场日志、历史故障、班组交接记录，也不能天然证明“我这句话来自哪一份文件”。

RAG 没有改变模型参数，也没有让模型本体获得新的长期记忆。它只是每次回答前临时检索证据，把相关证据放进上下文，让模型基于证据回答。所以它带来的核心提升不是智商，而是 grounding、traceability、auditability。

## 系统机制

典型 RAG 链路：

1. 收集来源：SOP、手册、日志、历史案例、政策、工艺说明。
2. 解析文档：把 PDF、网页、表格、文本转成可处理内容。
3. 切块：把长文档切成可检索片段。
4. 向量化：把文本变成 embedding。
5. 建索引：存入 vector database 或 hybrid search index。
6. 查询改写：把用户问题转成适合检索的问题。
7. 检索与重排：找出最可能相关的证据片段。
8. 生成回答：LLM 只基于证据和指令输出。
9. 引用与校验：显示来源，并检查答案是否真的被来源支持。

## 为什么“有来源”重要

没有 RAG 时，模型可能回答：可能是窑内通风不足、燃料波动或传感器异常。

有 RAG 时，系统应回答：根据 `SOP 3.2`、`2026-04-12 报警日志`、`维修案例 #184`，相似模式通常与二次风量下降和燃料波动有关；但当前传感器实时值不在 RAG 文档中，需要查询 time-series 数据确认。

区别在于：第一种是泛化猜测；第二种把结论连接到具体证据，并说明哪些部分还缺证据。

## 为什么“可检查”重要

可检查意味着人或程序可以问：

- 证据片段是否真的存在？
- 证据是否过期？
- 引用是否支持结论？
- 是否遗漏了更权威的资料？
- 这个回答是事实、推断，还是建议？
- 如果需要当前温度/压力/转速，是否错误地拿静态文档代替实时数据？

这使 AI 输出从“听起来很像对的”变成“可以被审计和改进的系统输出”。

## 为什么“可追责”重要

工业场景里，错误建议可能造成停机、质量事故、安全风险、成本损失或责任纠纷。可追责不是为了找人背锅，而是为了定位系统故障来源：

| 错误类型 | 追责/修复位置 |
|---|---|
| 找错文档 | retrieval / reranking 问题 |
| 文档过期 | source governance 问题 |
| 引用正确但结论过度推断 | generation / prompt / eval 问题 |
| 数据本来就缺 | knowledge coverage 问题 |
| 应该查实时数据却查了文档 | routing / tool selection 问题 |
| 答案没有拒答 | safety / fail-closed policy 问题 |

没有 RAG，你很难知道错在哪里；有 RAG，你至少可以沿着证据链排查。

## 对 Industrial AI SaaS 的用法

判断是否用 RAG，先问：这个工作流的价值是不是来自“快速找到并解释内部证据”？

强 RAG 场景：

- 操作员问某个报警过去怎么处理。
- 维修工程师查设备手册和历史工单。
- 新员工学习安全 SOP。
- PM 或工程师做事故复盘报告。
- 班组交接时快速理解历史异常。

不应 RAG-first 的场景：

- 查询当前温度、压力、电流、转速：用 time-series database / dashboard / API。
- 自动审批、派单、通知：用 workflow automation。
- 自动下控制指令：用 hard rules、权限、仿真、风控边界、审批或 fail-closed 机制。
- 燃料/产量优化：用 simulation / optimization / control model。
- 精确统计报表：用 SQL / structured query。

## ROI 判断

RAG 的 ROI 来自：

- 减少资料搜索时间。
- 减少专家重复答疑。
- 降低新人培训成本。
- 缩短故障排查时间。
- 把隐性经验沉淀为可复用证据链。
- 让 AI 输出更容易被工程、管理、合规接受。

但如果一个场景没有可靠资料、没有高频搜索、没有审计需求、没有明确的工作流价值，RAG 可能只是昂贵 demo。
